### 单线程和事件循环

`js是单线程的`

单线程的特点: 任务依次执行，后面的必须等前面的任务完成后，才能执行下一个任务。

* 事件循环: 

  * 任务分为同步任务和异步任务,异步任务又分为微任务和宏任务

  * 执行顺序是  首先同步任务执行   =>   微任务   =>  宏任务

  * 常见的宏任务：script（整体代码）、setTimeout、setInterval、I/O、setImmedidate，ajax

    常见的微任务：process.nextTick、MutationObserver、Promise.then catch finally

  其实宏任务速度要高于微任务,只是宏任务被放到下次事件队列中执行(早于下次的同步任务执行),

  所以便于好记 :  同步任务 => 微任务  => 宏任务

---

### 浏览器事件循环 vs Node.js 事件循环

#### 浏览器事件循环

结构简单，每轮循环固定为：

```
执行一个宏任务 → 清空所有微任务 → 渲染 → 下一个宏任务 ...
```

微任务队列只有一个，每次宏任务结束后必须清空。

```js
console.log('1')                          // 同步
setTimeout(() => console.log('2'), 0)     // 宏任务
Promise.resolve().then(() => console.log('3'))  // 微任务

// 输出：1 → 3 → 2
```

---

#### Node.js 事件循环

Node.js 基于 libuv，事件循环分为 **6 个阶段**，按顺序轮转：

```
   ┌─────────────────────────┐
   │         timers          │  执行 setTimeout / setInterval 回调
   ├─────────────────────────┤
   │     pending callbacks   │  上一轮延迟的 I/O 回调
   ├─────────────────────────┤
   │       idle, prepare     │  内部使用
   ├─────────────────────────┤
   │          poll           │  获取新的 I/O 事件（核心阶段）
   ├─────────────────────────┤
   │          check          │  执行 setImmediate 回调
   ├─────────────────────────┤
   │     close callbacks     │  关闭事件回调（如 socket.on('close')）
   └─────────────────────────┘
```

**每个阶段切换前**都会清空微任务队列（process.nextTick 优先于 Promise.then）。

---

#### 6 个阶段怎么记？

口诀：**"定挂闲轮查关"**

```
定 - timers          （定时器：setTimeout/setInterval）
挂 - pending         （挂起的回调：上轮未处理的 I/O 错误）
闲 - idle/prepare    （闲置：内部使用，面试可忽略）
轮 - poll            （轮询：等待新 I/O 事件，核心阶段）
查 - check           （查看：setImmediate）
关 - close           （关闭：socket.close 等）
```

每个阶段有自己独立的回调队列：

```
timers 队列   → 存 setTimeout/setInterval 到期的回调
pending 队列  → 存上一轮推迟的 I/O 错误回调
poll 队列     → 存 I/O 完成的回调（fs.readFile 等）
check 队列    → 存 setImmediate 的回调
close 队列    → 存关闭事件的回调
```

---

#### 微任务有几个队列？

有 **两个独立队列**，不是一个：

```
nextTick 队列  → 专门存 process.nextTick 的回调（优先级更高）
Promise 队列   → 存 Promise.then/catch/finally 的回调
```

清空规则是**循环检查直到全空**，不是各清一次就进入下一阶段：

```
清空 nextTick 队列
  → 清空 Promise 队列
    → Promise 执行中又加了 nextTick？
      → 立刻再清 nextTick 队列（插队！）
        → 再清 Promise 队列
          → 直到两个队列都空了
            → 才进入下一阶段
```

完整执行顺序图：

```
┌─────────────────────────────────────────┐
│              执行同步代码                │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│  清空 nextTick 队列 → 清空 Promise 队列  │ ← 每个阶段切换前都执行这步
└──────────────────┬──────────────────────┘
                   ▼
       ┌───────────────────────┐
       │       timers          │
       ├───────────────────────┤
       │  微任务（nextTick→Promise）│
       ├───────────────────────┤
       │       poll            │
       ├───────────────────────┤
       │  微任务（nextTick→Promise）│
       ├───────────────────────┤
       │       check           │
       ├───────────────────────┤
       │  微任务（nextTick→Promise）│
       └───────────────────────┘
```

#### 核心区别对比

| | 浏览器 | Node.js |
|---|---|---|
| 宏任务队列 | 1 个队列 | 6 个阶段，各有独立队列 |
| 微任务时机 | 每个宏任务后清空 | 每个阶段切换前清空 |
| 独有 API | `requestAnimationFrame` | `setImmediate`、`process.nextTick` |
| nextTick | 无 | 优先级高于所有微任务，最先执行 |

---

#### Node.js 特有的优先级顺序

```
process.nextTick  >  Promise.then  >  setImmediate  >  setTimeout
```

```js
setTimeout(() => console.log('setTimeout'), 0)
setImmediate(() => console.log('setImmediate'))
Promise.resolve().then(() => console.log('Promise'))
process.nextTick(() => console.log('nextTick'))

// 输出：nextTick → Promise → setTimeout/setImmediate（后两者顺序不固定）
```

> `setTimeout` 和 `setImmediate` 的顺序在主模块中不固定（受系统计时器精度影响），但在 I/O 回调中 `setImmediate` 一定先于 `setTimeout`。

#### 面试常考题

**Q: process.nextTick 和 Promise.then 哪个先执行？**

`process.nextTick` 先。它有独立的 nextTick 队列，优先级高于其他所有微任务。

**Q: setImmediate 和 setTimeout(fn, 0) 哪个先？**

- 主模块中：顺序**不固定**，取决于系统计时器
- I/O 回调中：`setImmediate` **一定先**，因为 I/O 回调在 poll 阶段，结束后直接进入 check 阶段（setImmediate），timers 阶段要等下一轮

**Q: Promise 执行中加入了 nextTick，需要等下一阶段结束才执行吗？**

不需要，**立刻插队执行**。微任务清空是循环检查，nextTick 优先级永远高于 Promise，无论何时加入都会在下一个 Promise 回调执行前先被消费。

```js
Promise.resolve().then(() => {
  console.log('Promise 1')
  process.nextTick(() => console.log('nextTick 插队'))
})
Promise.resolve().then(() => {
  console.log('Promise 2')
})
// 输出：Promise 1 → nextTick 插队 → Promise 2
```

> 注意：如果 nextTick 里不断加新的 nextTick，会把 Promise 队列和后续阶段全部饿死（starve），所以不建议滥用 `process.nextTick`。
