# 组件化设计

## 什么是组件化？

将页面拆分为独立、可复用的单元，每个组件管理自己的结构（HTML）、样式（CSS）、逻辑（JS）。

核心目标：**高内聚、低耦合、可复用**

---

## 面试官常问角度

### Q1: 你是怎么划分组件的？

按照三个维度拆分：

```
1. 按职责划分
   ├── 展示型组件（Presentational）：只管渲染，数据由外部传入，无副作用
   └── 容器型组件（Container）：负责数据获取、状态管理，把数据传给展示组件

2. 按复用粒度划分
   ├── 基础组件（Button、Input、Modal）：通用 UI，不含业务逻辑
   ├── 业务组件（UserCard、OrderList）：组合基础组件，含业务逻辑
   └── 页面组件（HomePage、DetailPage）：路由级别，组合业务组件

3. 按变化频率划分
   └── 变化频繁的逻辑单独抽离，稳定的部分做成纯展示组件，减少不必要的重渲染
```

### Q2: 组件设计有哪些原则？

**单一职责原则**
> 一个组件只做一件事，功能越单一越好复用

```
❌ 一个组件既请求数据、又处理逻辑、又渲染 UI
✅ 拆成：数据层（hooks/store）+ 逻辑层（容器组件）+ 渲染层（展示组件）
```

**开放封闭原则**
> 对扩展开放，对修改封闭。通过 props/slots 暴露扩展点，而不是直接修改组件内部

```jsx
// ❌ 改需求就要改组件源码
<Button type="primary" />

// ✅ 通过 props 扩展，不改源码
<Button renderIcon={() => <Icon />} onClick={handler} />
```

**受控与非受控**
> 根据场景决定状态由外部控制（受控）还是组件内部维护（非受控）

```jsx
// 受控：状态在父组件，父组件完全掌控
<Input value={val} onChange={setVal} />

// 非受控：状态在组件内部，外部只关心最终结果
<Input defaultValue="hello" ref={inputRef} />
```

### Q3: 如何设计一个通用组件（以 Modal 为例）？

设计时需要考虑：

```
1. Props 设计
   - visible/open：控制显示隐藏（受控）
   - title：标题
   - onClose / onConfirm：事件回调
   - footer：自定义底部（插槽/render props）
   - width、zIndex：样式可配置

2. 插槽/children 设计
   - 默认插槽：弹窗内容区域
   - 具名插槽：header、footer 可自定义

3. 边界情况
   - 点击遮罩关闭（可配置是否开启）
   - ESC 键关闭
   - 滚动穿透问题
   - 多个 Modal 叠加时的 zIndex 管理

4. 无障碍（a11y）
   - role="dialog"、aria-modal、焦点锁定在 Modal 内
```

### Q4: 组件通信有哪些方式？

```
父 → 子：props
子 → 父：emit 事件 / 回调函数
兄弟组件：提升状态到共同父组件 / 事件总线
跨层级：Context（React）/ provide-inject（Vue）/ 全局状态管理（Pinia/Redux）
```

选择原则：**能用 props 解决就用 props，层级超过 3 层考虑 Context，全局共享状态用状态管理库**。

### Q5: 如何避免组件过度设计？

常见的过度设计问题：

```
❌ 一开始就抽象所有东西
❌ 为了"以后可能用到"加很多无用 props
❌ 组件拆得太细，props drilling 严重
```

正确做法：**三次法则**
> 同一段逻辑出现第三次，再考虑抽组件。一两次先 copy，等需求稳定再抽象。

### Q6: 性能角度怎么设计组件？

```
1. 减少不必要的渲染
   - React：memo、useMemo、useCallback
   - Vue：v-memo、computed

2. 合理拆分组件粒度
   - 频繁变化的状态下沉到子组件，避免父组件整体重渲染

3. 懒加载
   - 路由级别：React.lazy / Vue defineAsyncComponent
   - 组件级别：滚动到视口才渲染（IntersectionObserver）

4. 列表优化
   - 虚拟列表：只渲染可视区域内的条目（react-virtual / vue-virtual-scroller）
   - key 的正确使用：用唯一 id，不用 index（避免 diff 错误）
```

### Q7: Hooks / Composition API 和组件化的关系？

Hooks 是对**逻辑**的组件化，组件是对 **UI + 逻辑**的组件化。

```
组件复用 UI + 逻辑（但会引入 DOM 结构）
Hooks  只复用逻辑（不引入额外 DOM，更灵活）
```

```js
// 把请求逻辑抽成 hook，多个组件复用，不耦合 UI
function useUserInfo(id) {
  const [user, setUser] = useState(null)
  useEffect(() => {
    fetchUser(id).then(setUser)
  }, [id])
  return user
}
```

## 面试回答思路总结

遇到"如何设计组件"类问题，可以按以下结构回答：

```
1. 先说划分原则（展示/容器、基础/业务/页面）
2. 再说设计原则（单一职责、开放封闭、受控非受控）
3. 举一个具体组件例子（Modal / Form / Table）说明 props 设计
4. 补充性能考虑（memo、虚拟列表、懒加载）
5. 提到 Hooks 逻辑复用作为加分项
```
