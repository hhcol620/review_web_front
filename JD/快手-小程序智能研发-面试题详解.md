## 一、JS / TS 基础

### Q1: var / let / const 区别，变量提升原理？

**核心区别：**

| | var | let | const |
|--|-----|-----|-------|
| 作用域 | 函数作用域 | 块级作用域 | 块级作用域 |
| 变量提升 | 提升且初始化为 undefined | 提升但不初始化（TDZ） | 提升但不初始化（TDZ） |
| 重复声明 | 允许 | 不允许 | 不允许 |
| 重新赋值 | 允许 | 允许 | 不允许（引用不可变） |
| 挂载到 window | 是 | 否 | 否 |

**变量提升原理：**
JS 引擎在执行代码前有一个「编译阶段」，会扫描当前作用域内所有声明并提前登记：
- `var` 声明会被提升并初始化为 `undefined`
- `let/const` 声明会被提升但不初始化，从声明位置到代码顶部之间的区域称为**暂时性死区（TDZ）**，访问会抛 ReferenceError

```js
console.log(a) // undefined（var 提升+初始化）
console.log(b) // ReferenceError（TDZ）
var a = 1
let b = 2
```

### Q2: 原型链是什么？如何实现继承？

**原型链：**
每个对象都有 `__proto__` 属性指向其构造函数的 `prototype`，访问属性时沿着这条链向上查找，直到 `null` 为止。

```
obj.__proto__ === Constructor.prototype
Constructor.prototype.__proto__ === Object.prototype
Object.prototype.__proto__ === null
```

**常见继承方式对比：**

```js
// 1. 原型链继承 —— 引用类型属性被共享，问题最大
Child.prototype = new Parent()

// 2. 借用构造函数 —— 解决共享问题，但无法继承原型方法
function Child() { Parent.call(this) }

// 3. 组合继承（最常用的 ES5 方案）
function Child() { Parent.call(this) }
Child.prototype = Object.create(Parent.prototype)
Child.prototype.constructor = Child

// 4. ES6 class（本质是上面的语法糖）
class Child extends Parent {
  constructor() { super() }
}
```

### Q3: 事件循环、宏任务、微任务？

**执行顺序：**
1. 执行同步代码（调用栈）
2. 调用栈清空后，清空**微任务队列**（全部执行完）
3. 取一个**宏任务**执行
4. 再次清空微任务队列
5. 重复 3-4

**常见分类：**

| 微任务 | 宏任务 |
|--------|--------|
| Promise.then/catch/finally | setTimeout / setInterval |
| queueMicrotask | MessageChannel |
| MutationObserver | I/O、UI 渲染 |
| async/await（await 后面的代码） | requestAnimationFrame |

```js
console.log('1')
setTimeout(() => console.log('2'), 0)
Promise.resolve().then(() => console.log('3'))
console.log('4')
// 输出：1 4 3 2
```

**常见面试考题：输出顺序预测**

```js
// 题一
async function async1() {
  console.log('async1 start')       // 2
  await async2()
  console.log('async1 end')         // 6
}
async function async2() {
  console.log('async2')             // 3
}
console.log('script start')         // 1
setTimeout(() => console.log('setTimeout'), 0) // 8
async1()
new Promise(resolve => {
  console.log('promise1')           // 4
  resolve()
}).then(() => console.log('promise2')) // 7
console.log('script end')           // 5
// 输出：script start → async1 start → async2 → promise1 → script end → async1 end → promise2 → setTimeout
```

> **关键点：** `await async2()` 相当于 `Promise.resolve(async2()).then(后续代码)`，await 之后的代码是微任务，但 `async2()` 函数体本身是同步执行的。

```js
// 题二：微任务中嵌套微任务
Promise.resolve()
  .then(() => {
    console.log(1)                  // 第一轮微任务
    return Promise.resolve(2)       // 注意：这里多等一轮！
  })
  .then(v => console.log(v))        // 第三轮微任务才执行

Promise.resolve()
  .then(() => console.log(3))       // 第一轮微任务
  .then(() => console.log(4))       // 第二轮微任务
// 输出：1 3 4 2
```

> **关键点：** `.then` 里 `return Promise.resolve(x)` 会额外多消耗两个微任务 tick（规范要求），所以 2 比 4 晚输出。

### Q4: Promise 原理 / 手写 Promise.all

**Promise.all 手写：**

```js
function promiseAll(promises) {
  return new Promise((resolve, reject) => {
    const results = []
    let count = 0
    promises.forEach((p, i) => {
      Promise.resolve(p).then(val => {
        results[i] = val
        count++
        if (count === promises.length) resolve(results)
      }).catch(reject)
    })
  })
}
```

**Promise.allSettled 手写：**

```js
function promiseAllSettled(promises) {
  return Promise.all(promises.map(p =>
    Promise.resolve(p)
      .then(value => ({ status: 'fulfilled', value }))
      .catch(reason => ({ status: 'rejected', reason }))
  ))
}
```

### Q6: interface 和 type 的区别？

**核心区别对比：**

| | interface | type |
|--|-----------|------|
| 声明合并 | 支持（同名自动合并） | 不支持，同名报错 |
| 扩展方式 | `extends` | `&`（交叉类型） |
| 联合类型 | 不支持 | 支持（`A \| B`） |
| 基本类型别名 | 不支持 | 支持（`type ID = string`） |
| 元组/映射类型 | 不支持 | 支持 |
| 计算属性 | 不支持 | 支持 |

**声明合并（interface 独有）：**
```ts
interface User { name: string }
interface User { age: number }
// 自动合并为 { name: string; age: number }
// 常用于扩展第三方库的类型定义
```

**扩展写法对比：**
```ts
// interface 用 extends
interface Animal { name: string }
interface Dog extends Animal { bark(): void }

// type 用交叉类型 &
type Animal = { name: string }
type Dog = Animal & { bark(): void }
```

**type 独有能力：**
```ts
// 联合类型
type Status = 'loading' | 'success' | 'error'

// 基本类型别名
type ID = string | number

// 条件类型
type IsArray<T> = T extends any[] ? true : false

// 映射类型
type Readonly<T> = { readonly [K in keyof T]: T[K] }
```

**如何选择：**
- 定义**对象结构/类的契约** → 优先用 `interface`（语义更清晰，支持合并）
- 需要**联合类型、交叉类型、工具类型** → 用 `type`
- 扩展第三方库类型 → 必须用 `interface`（利用声明合并）

### Q7: 闭包是什么？会导致内存泄漏吗？

**定义：** 函数可以访问其定义时所在词法作用域中的变量，即使该函数在其他地方执行。

**内存泄漏场景：**

```js
// 泄漏：largData 一直被闭包引用无法回收
function createLeak() {
  const largeData = new Array(1000000)
  return function() {
    console.log(largeData.length)
  }
}
const leak = createLeak()

// 解决：手动置 null
function createSafe() {
  let largeData = new Array(1000000)
  const len = largeData.length
  largeData = null  // 释放大对象
  return function() {
    console.log(len)
  }
}
```

## 二、框架原理

### Q7: Vue3 响应式原理？Proxy vs Object.defineProperty？

**Vue2 Object.defineProperty 的问题：**
- 无法检测对象属性的新增/删除（需要 `$set`）
- 无法检测数组下标修改（需要重写数组方法）
- 需要递归遍历对象所有属性，初始化性能差

**Vue3 Proxy 的优势：**

```js
function reactive(obj) {
  return new Proxy(obj, {
    get(target, key, receiver) {
      track(target, key)  // 依赖收集
      const res = Reflect.get(target, key, receiver)
      // 懒递归：访问时才代理嵌套对象
      return typeof res === 'object' ? reactive(res) : res
    },
    set(target, key, value, receiver) {
      const res = Reflect.set(target, key, value, receiver)
      trigger(target, key)  // 触发更新
      return res
    }
  })
}
```

- 可以拦截属性新增/删除（`deleteProperty`）
- 可以拦截数组任何操作
- 懒代理，只有访问到才递归，性能更好

### Q8: React Fiber 是什么？解决了什么问题？

**问题背景：**
React 16 之前的 Stack Reconciler 是同步递归的，一旦开始就无法中断，如果组件树很深，JS 线程长时间占用，导致页面卡顿（掉帧）。

**Fiber 方案：**
把渲染工作拆成小单元（Fiber 节点），每个单元执行完后检查是否有更高优先级任务，有则中断当前工作，先处理高优先级任务（时间切片）。

**Fiber 核心字段：**
```js
{
  type,          // 组件类型
  stateNode,     // 对应的真实 DOM 或组件实例
  return,        // 父节点
  child,         // 第一个子节点
  sibling,       // 下一个兄弟节点
  alternate,     // 双缓冲：指向上一次的 Fiber 树
  effectTag,     // 需要执行的操作（插入/更新/删除）
  lanes,         // 优先级
}
```

**两个阶段：**
- **Render 阶段**（可中断）：遍历 Fiber 树，找出需要变更的节点，打 effectTag
- **Commit 阶段**（不可中断）：把变更一次性同步到真实 DOM

### Q9: Diff 算法，key 为什么不用 index？

**Diff 策略（同层比较）：**
1. 新旧节点类型不同 → 直接销毁重建
2. 类型相同 → 复用节点，只更新属性
3. 子节点列表 → 通过 key 做映射，找到可复用的节点

**index 作为 key 的问题：**

```jsx
// 列表 [A, B, C]，删除 A 后变成 [B, C]
// 用 index 作为 key：
// 旧: A(key=0) B(key=1) C(key=2)
// 新: B(key=0) C(key=1)
// React 认为 key=0 的节点还在，只是内容变了，会复用 A 的 DOM 更新内容
// 如果组件有内部状态（如 input 的值），会错乱
```

**正确做法：** 用数据的唯一标识（id）作为 key。

## 三、Node.js

### Q11: Node.js 事件循环与浏览器的区别？

#### 浏览器事件循环（简单）

```
同步代码 → 清空微任务 → 取一个宏任务 → 清空微任务 → 取一个宏任务 → ...
```

每执行完一个宏任务，立即清空所有微任务，然后再取下一个宏任务。

---

#### Node.js 事件循环（libuv，6 个阶段）

```
┌──────────────────────────┐
│  timers                  │  执行 setTimeout / setInterval 到期的回调
├──────────────────────────┤
│  pending callbacks       │  执行上一轮遗留的 I/O 错误回调
├──────────────────────────┤
│  idle, prepare           │  Node 内部使用
├──────────────────────────┤
│  poll                    │  取新的 I/O 事件并执行，没有任务时在此等待
├──────────────────────────┤
│  check                   │  执行 setImmediate 回调
├──────────────────────────┤
│  close callbacks         │  执行 close 事件（如 socket.on('close')）
└──────────────────────────┘
       ↑__________________|  每轮结束回到 timers
```

**微任务在哪里执行？**
- **Node v11 之前**：每个阶段全部执行完，才统一清空微任务队列
- **Node v11 之后**：与浏览器对齐，每个宏任务/回调执行完立即清空微任务

---

#### Node.js 独有：process.nextTick 和 setImmediate

| API | 队列 | 执行时机 |
|-----|------|---------|
| `process.nextTick` | nextTick 队列 | 当前操作完成后，**任何微任务之前** |
| `Promise.then` | 微任务队列 | nextTick 队列清空后 |
| `setImmediate` | check 阶段 | poll 阶段之后（本轮循环末尾） |
| `setTimeout(fn, 0)` | timers 阶段 | 下一轮循环开始 |

**优先级从高到低：**
```
process.nextTick > Promise.then > setImmediate ≈ setTimeout(fn, 0)
```

---

#### 常见面试考题

**题一：nextTick vs Promise 优先级**
```js
Promise.resolve().then(() => console.log('promise'))
process.nextTick(() => console.log('nextTick'))
// 输出：nextTick → promise
// 原因：nextTick 队列优先级高于微任务队列
```

**题二：setImmediate vs setTimeout 顺序**
```js
// 在主模块中（非 I/O 回调里）
setTimeout(() => console.log('timeout'), 0)
setImmediate(() => console.log('immediate'))
// 输出：不确定！取决于事件循环启动耗时
// timeout 可能先，也可能 immediate 先
```

```js
// 在 I/O 回调中
const fs = require('fs')
fs.readFile('./file', () => {
  setTimeout(() => console.log('timeout'), 0)
  setImmediate(() => console.log('immediate'))
})
// 输出：immediate → timeout（确定）
// 原因：I/O 回调在 poll 阶段执行，下一个阶段就是 check（setImmediate）
//       timers 阶段要等下一轮循环
```

**题三：nextTick 递归会怎样**
```js
function loop() {
  process.nextTick(loop)
}
loop()
// 死循环！nextTick 递归会一直占用 nextTick 队列，I/O 永远得不到执行
// setImmediate 递归不会有这个问题（每轮循环才执行一次）
```

---

#### 一句话总结区别

| 对比项 | 浏览器 | Node.js |
|--------|--------|---------|
| 事件循环结构 | 简单的宏/微任务队列 | 6 个阶段的循环（libuv） |
| 独有 API | — | `process.nextTick`、`setImmediate` |
| 微任务时机 | 每个宏任务后立即清空 | v11 后与浏览器一致 |
| I/O 模型 | 浏览器处理 | libuv 线程池处理 |

### Q12: Koa 洋葱模型是什么？手写中间件核心？

**洋葱模型：** 请求依次经过每个中间件，响应时再反向经过，形成洋葱状的调用链。

```js
// 手写 compose（Koa 核心）
function compose(middlewares) {
  return function(ctx) {
    function dispatch(i) {
      const fn = middlewares[i]
      if (!fn) return Promise.resolve()
      return Promise.resolve(fn(ctx, () => dispatch(i + 1)))
    }
    return dispatch(0)
  }
}

// 使用
const mid1 = async (ctx, next) => {
  console.log('1 进')
  await next()
  console.log('1 出')
}
const mid2 = async (ctx, next) => {
  console.log('2 进')
  await next()
  console.log('2 出')
}
// 输出：1进 2进 2出 1出
```

### Q13: CommonJS 和 ESM 的区别？

| | CommonJS | ESM |
|--|----------|-----|
| 加载时机 | 运行时（动态） | 编译时（静态分析） |
| 导出值 | 值的拷贝 | 值的引用（live binding） |
| this | module | undefined |
| 循环依赖 | 拿到不完整的 exports | 可以处理，但要注意顺序 |
| Tree Shaking | 不支持 | 支持 |

```js
// CJS 导出值拷贝
let count = 0
module.exports = { count, add: () => count++ }
// 引用方拿到的 count 是 0 的拷贝，不会随 add() 变化

// ESM live binding
export let count = 0
export const add = () => count++
// 引用方拿到的 count 是实时的
```

## 四、工程化

### Q14: Webpack 构建流程？Loader 和 Plugin 的区别？

**构建流程：**
1. 读取配置，初始化参数
2. 创建 Compiler，触发 `environment` 等初始化 hooks
3. 从 entry 开始递归解析依赖，构建模块依赖图
4. 对每个模块使用对应 **Loader** 转换（如 ts-loader、babel-loader）
5. 将模块组装成 **Chunk**
6. 输出到 dist 目录

**Loader vs Plugin：**

| | Loader | Plugin |
|--|--------|--------|
| 作用 | 转换文件内容（A → B） | 扩展构建能力，贯穿整个生命周期 |
| 本质 | 函数，接收文件内容返回转换结果 | 类，通过 hooks 介入构建流程 |
| 示例 | babel-loader、css-loader | HtmlWebpackPlugin、MiniCssExtractPlugin |

```js
// 最简单的 Loader
module.exports = function(source) {
  return source.replace(/console\.log\(.*?\)/g, '')
}

// 最简单的 Plugin
class MyPlugin {
  apply(compiler) {
    compiler.hooks.emit.tapAsync('MyPlugin', (compilation, callback) => {
      // 在输出文件前做点什么
      callback()
    })
  }
}
```

### Q15: Vite 为什么比 Webpack 快？

**开发阶段：**
- Webpack：启动时打包所有模块，生成 bundle 后启动 devServer
- Vite：直接启动 devServer，利用浏览器原生 ESM，按需编译（请求哪个文件才编译哪个）

**HMR（热更新）：**
- Webpack：重新打包受影响的整个 chunk
- Vite：只失效变更的模块本身，精确热替换

**预构建：**
Vite 用 esbuild（Go 编写）预构建 node_modules 中的 CJS 依赖，速度比 babel 快 10-100 倍。

**生产构建：**
Vite 生产环境用 Rollup 打包，Tree Shaking 更彻底，产物更小。

### Q16: Tree Shaking 原理？什么情况下失效？

**原理：** 基于 ESM 静态导入/导出分析，编译时就能确定哪些导出被使用，打包时删除未使用的代码（Dead Code Elimination）。

**失效场景：**
1. 使用 CommonJS（require 是动态的，编译时无法分析）
2. 导入了有副作用的模块（如修改全局变量），需要在 `package.json` 中配置 `"sideEffects": false`
3. `export default` 导出对象整体（无法按需分析内部属性）
4. Babel 把 ESM 转成了 CJS（配置 `@babel/preset-env` 的 `modules: false` 解决）

```json
// package.json
{
  "sideEffects": ["*.css", "./src/polyfills.js"]  // 指定有副作用的文件
}
```

### Q17: 前端性能优化手段？

**加载阶段：**
- 代码分割（路由懒加载）+ preload/prefetch
- 图片懒加载、WebP 格式、CDN
- HTTP/2 多路复用
- Gzip/Brotli 压缩
- 缓存策略（强缓存 + 协商缓存）

**运行时：**
- 虚拟列表（长列表只渲染可视区域）
- `requestAnimationFrame` 替代 setTimeout 做动画
- Web Worker 处理耗时计算
- 防抖/节流
- 避免强制同步布局（读写 DOM 分离）

**渲染：**
- 减少重排（reflow）：批量修改 DOM、使用 `transform` 替代 `top/left`
- 使用 `will-change` 开启 GPU 合成层（谨慎使用）
- SSR / SSG 优化首屏

## 五、小程序

### Q18: 小程序双线程架构？

**两个线程：**
- **渲染层**（WebView）：负责页面渲染，运行 WXML/WXSS
- **逻辑层**（JSCore）：运行 JS 逻辑，无法直接操作 DOM

**通信流程：**
```
逻辑层 setData(data)
  → Native Bridge（序列化数据）
    → 渲染层接收数据，更新视图
```

**为什么这样设计？**
- 安全性：逻辑层无法直接操作 DOM，防止开发者绕过审核注入恶意代码
- 稳定性：逻辑层崩溃不影响渲染层，页面不会白屏

**性能问题：** `setData` 数据量过大会导致通信阻塞，优化方案：
- 只传变更的字段，使用路径更新 `setData({ 'list[0].name': 'foo' })`
- 避免频繁 setData，批量合并更新
- 图片/长列表等不放在 data 里

### Q20: 手写防抖 / 节流

```js
// 防抖：最后一次触发后 delay 执行
function debounce(fn, delay) {
  let timer = null
  return function(...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}

// 节流：每隔 interval 最多执行一次
function throttle(fn, interval) {
  let last = 0
  return function(...args) {
    const now = Date.now()
    if (now - last >= interval) {
      last = now
      fn.apply(this, args)
    }
  }
}
```

### Q21: 手写深拷贝

```js
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj
  if (obj instanceof Date) return new Date(obj)
  if (obj instanceof RegExp) return new RegExp(obj)

  // 处理循环引用
  if (map.has(obj)) return map.get(obj)

  const clone = Array.isArray(obj) ? [] : {}
  map.set(obj, clone)

  for (const key of Reflect.ownKeys(obj)) {
    clone[key] = deepClone(obj[key], map)
  }
  return clone
}
```

### Q22: 手写 EventEmitter（发布订阅）

```js
class EventEmitter {
  constructor() {
    this.events = {}
  }
  on(event, fn) {
    if (!this.events[event]) this.events[event] = []
    this.events[event].push(fn)
    return this
  }
  emit(event, ...args) {
    (this.events[event] || []).forEach(fn => fn(...args))
    return this
  }
  off(event, fn) {
    this.events[event] = (this.events[event] || []).filter(f => f !== fn)
    return this
  }
  once(event, fn) {
    const wrapper = (...args) => {
      fn(...args)
      this.off(event, wrapper)
    }
    return this.on(event, wrapper)
  }
}
```

### Q23: 什么是 RAG？前端如何接入？

**RAG（检索增强生成）：** 在调用 LLM 之前，先从知识库中检索相关内容，把检索结果作为上下文一起发给 LLM，解决 LLM 知识截止和幻觉问题。

**前端接入 LLM 流式输出（SSE）：**

```js
async function streamChat(prompt) {
  const response = await fetch('/api/chat', {
    method: 'POST',
    body: JSON.stringify({ prompt }),
    headers: { 'Content-Type': 'application/json' }
  })

  const reader = response.body.getReader()
  const decoder = new TextDecoder()

  while (true) {
    const { done, value } = await reader.read()
    if (done) break
    const chunk = decoder.decode(value)
    // 解析 SSE 数据
    const lines = chunk.split('\n').filter(line => line.startsWith('data:'))
    for (const line of lines) {
      const data = JSON.parse(line.slice(5))
      // 追加到界面
      appendText(data.content)
    }
  }
}
```
